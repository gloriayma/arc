# Step 3.5 (chr22 pilot) — STAR index on the donor consensus

Goal: build a chr22 STAR index from `joxm.chr22.consensus_snps.fa` so step 5 can splice-align cell FASTQs against the personalised donor genome. Splice-junction-aware via GENCODE v19 chr22.

## Final command (works, ~20 s)

```fish
cd data/hipsci/joxm/chr22

STAR --runMode genomeGenerate \
     --genomeDir STAR_index_joxm_chr22 \
     --genomeFastaFiles joxm.chr22.consensus_snps.fa \
     --sjdbGTFfile gencode.v19.chr22.gtf \
     --sjdbOverhang 124 \
     --genomeSAindexNbases 11 \
     --runThreadN 4
```

Produces `STAR_index_joxm_chr22/` (~366 MB).

## Inputs

- `joxm.chr22.consensus_snps.fa` (50 MB) — chr22 with 16,418 HOM-ALT SNVs baked in (from step 3)
- `gencode.v19.chr22.gtf` — chr22 lines extracted from `gencode.v19.annotation.nochr.gtf.gz`, 58,792 lines, 1,263 genes, 7,674 splice junctions in the GENCODE annotation

## Parameters and why

| Flag | Value | Rationale |
|---|---|---|
| `--runMode genomeGenerate` | — | index build mode (vs `alignReads` later) |
| `--sjdbGTFfile` | chr22 GTF | passes splice-junction annotations so STAR is splice-aware at alignment time |
| `--sjdbOverhang` | **124** | (read_length − 1). Cuomo CRAMs are 125 bp paired-end (verified by streaming first reads of `24722_3%23101.cram` — `HS36_24722:3:2113:3089:64908` SEQ length = 125). Common defaults of `99` or `100` would still work but slightly worse for splice detection at read edges |
| `--genomeSAindexNbases` | **11** | STAR docs: `min(14, log2(genomeLength)/2 - 1)`. chr22 = 51,304,566 bp → `log2(5.13e7)/2 - 1 ≈ 11.8`. Default `14` over-indexes a small genome and balloons memory for no gain |
| `--runThreadN` | 4 | local M-series core count; build takes 13 s wall-clock for chr22 |

## Outputs (in `STAR_index_joxm_chr22/`)

| File | Size | Purpose |
|---|---|---|
| `Genome` | 51 MB | packed binary copy of the donor chr22 sequence |
| `SA` | 290 MB | suffix array — main index data structure for read lookup |
| `SAindex` | 23 MB | suffix array's lookup table (depth controlled by `--genomeSAindexNbases`) |
| `sjdbList.fromGTF.out.tab` | 205 KB | 7,674 splice junctions extracted from the chr22 GTF |
| `sjdbList.out.tab` | 172 KB | union of GTF junctions + STAR-inferred (here just the GTF set since no novel junctions yet) |
| `sjdbInfo.txt` | 195 KB | junction metadata for runtime use |
| `exonGeTrInfo.tab` / `exonInfo.tab` / `geneInfo.tab` / `transcriptInfo.tab` | — | gene/exon/transcript tables (1,263 genes, e.g. `ENSG00000233866.1 LA16c-4G1.3 lincRNA`) |
| `chrNameLength.txt` | 12 B | `22\t51304566` — single chromosome in this pilot index |
| `Log.out` | 4.6 KB | full STAR build log |
| `genomeParameters.txt` | 692 B | recorded parameters; STAR will refuse to align reads if you change critical params at runtime |

## Gotcha: the no-chr GTF had to be rebuilt with `awk -F'\t'`

The version of `gencode.v19.annotation.nochr.gtf.gz` produced in step 2 was **silently corrupted**. The remap awk script (`awk 'BEGIN{OFS="\t"} ... sub(/^chr/, "", $1)`) ran without `-F'\t'`, so awk used its default whitespace-splitting field separator. The GTF spec uses tabs only between the 9 mandatory fields, but the 9th attribute field contains spaces (`gene_id "ENSG..."; transcript_id "..."`). awk therefore split each line into ~28 fields, then on print rejoined them with `OFS="\t"`, replacing every space in the attribute column with a tab.

How I noticed: extracted chr22 lines, ran STAR, but first sanity-printed the GTF — fields 9 onward were `gene_id` `"ENSG..."`; `transcript_id` `"..."` etc. all tab-separated. STAR would have failed to parse the attributes.

Fix: rebuild with explicit tab separator.

```fish
zcat < gencode.v19.annotation.gtf.gz | \
  awk -F'\t' 'BEGIN{OFS="\t"} /^#/ {print; next} {
    sub(/^chr/, "", $1);
    if ($1 == "M") $1 = "MT";
    print
  }' | gzip > gencode.v19.annotation.nochr.gtf.gz
```

After fix, field count is `NF=9` with `awk -F'\t'`, attribute strings are intact, and STAR parses the GTF cleanly (build log shows `processing annotations GTF` + 7,674 splice junctions). Same fix applied to `gencode.v19.chr22.gtf`.

This bug doesn't depend on which downstream tool reads the GTF — anything that expects standard GTF will be misled by the corrupted version. If we ever revisit step 2 doc, the awk snippet there should be updated too.

## Gotcha: the brew formula `star` is the wrong STAR

`brew install star` gets `schilytools/star`, a tar-replacement archive utility — not the RNA-seq aligner. Reinstall:

```fish
brew uninstall star
brew install rna-star
```

Installs `STAR 2.7.11b` (Alexander Dobin's aligner) at `/opt/homebrew/bin/STAR`. The two formulae conflict on the binary name `STAR`, so they can't coexist.
