Before downloading anything large, we grab two small files that tell us which cells exist and how to find them. The whole pipeline is organized around grouping cells by donor — every cell from the same donor gets aligned to that donor's personalized genome.

This file maps every cell ID to its donor, time point, and experiment. It's the central index for the dataset.

Why group by donor? Each donor gets their own N-masked personalized reference genome. A cell from donor bezi must be aligned to bezi's FASTA (with bezi's het sites N-masked and WASP VCF), not lexy's. The metadata tells us which cells belong to which donor.

Download reference files

hs37d5 reference genome HipSci FTP · ~3 GB

A FASTA file is the simplest possible genome format: a header line starting with > giving the chromosome name, followed by lines of the DNA sequence.

hs37d5.fa FASTA
>1 ← chromosome 1 (GRCh37 uses numbers, not "chr1") NNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNN NNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNN TAACCCTAACCCTAACCCTAACCCTAACCCTAACCCTAACCCTAACCCT TAACCCTAACCCTAACCCTAACCCTAACCCTAACCCTAACCCTAACCCT GCGCGAAAGCGCTCAGGCGCGAAAACGCCTAACCCAAACCCAAACCC ... 249,250,621 bases for chr1 alone ... file continues through chr2, chr3, ... chrX, chrY, MT, decoys ... total: ~3.2 GB uncompressed >2 NNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNN ...
Why hs37d5 specifically? The CRAMs from ENA were compressed against this exact reference genome. CRAM is a format that stores only the differences from a reference, so you need the identical reference to reconstruct the original reads. Using a different genome (even GRCh38) would produce corrupted output.

GENCODE v19 gene annotation GENCODE · 35 MB

A GTF (Gene Transfer Format) file lists the coordinates of every known gene, transcript, and exon. STAR reads this to know where splice junctions are, so it can correctly align reads that span exon-exon junctions.

gencode.v19.annotation.gtf GTF
# Each row is one feature. Columns: chr, source, type, start, end, score, strand, frame, attributes 1 HAVANA gene 11869 14412 . + . gene_id "ENSG00000223972.4"; gene_name "DDX11L1"; 1 HAVANA transcript 11869 14409 . + . gene_id "ENSG00000223972.4"; transcript_id "ENST00000456328.2"; 1 HAVANA exon 11869 12227 . + . exon_id "ENSE00002234944.1"; exon_number "1"; 1 HAVANA exon 12613 12721 . + . exon_id "ENSE00003582793.1"; exon_number "2"; 1 HAVANA exon 13221 14409 . + . exon_id "ENSE00002312635.1"; exon_number "3"; ... ~2.3M rows for all genes/transcripts/exons

The exon rows are what matter most. STAR uses exon boundaries to know where it's valid for a read to "jump" — if a 75 bp read starts in exon 1 and ends in exon 2, STAR needs to know that the gap between them (the intron) is a legitimate splice junction, not a sequencing error.

3
Build per-donor N-masked FASTA (the key step)

This is what makes this pipeline different from a standard RNA-seq alignment. Instead of aligning all 36,000 cells to the same reference genome, each donor gets their own personalized reference built from their actual genetic variants. We use an N-masked single FASTA rather than two separate haplotype FASTAs — heterozygous positions are replaced with N so the aligner cannot favour either allele. Haplotype splitting happens later with SNPsplit after alignment.

Download imputed joint BCF HipSci FTP · ~6 GB

A BCF is a binary VCF — the standard format for listing genetic variants. A VCF (Variant Call Format) file has one row per variant, recording what the DNA looks like at that position for each sample. The critical feature here is phasing.

🔀 Unphased genotype: 0/1

We know this person is heterozygous (has one reference allele and one alternate allele) but we don't know which chromosome copy has which allele. The slash / means "order unknown".

✅ Phased genotype: 0|1

The pipe | means the order matters. The left allele (0 = reference) is on haplotype 1; the right allele (1 = alternate) is on haplotype 2. We know which chromosome copy has which allele.

chr1.bcf → per_donor/HPSI0114i-bezi_1.vcf.gz (after extraction) VCF
##fileformat=VCFv4.1 ##reference=hs37d5 ##phasing=SHAPEIT2 ##FORMAT=<ID=GT,Type=String,Description="Phased Genotype"> ##FORMAT=<ID=GP,Type=Float,Description="Genotype Probabilities"> #CHROM POS ID REF ALT QUAL FILTER INFO FORMAT HPSI0114i-bezi_1 ────────────────────────────────────────────────────────────────────────── 1 752566 . A G . PASS . GT:GP 0|1:0.01,0.98,0.01 ↑↑↑ hap1=REF(A), hap2=ALT(G) 1 754192 . C T . PASS . GT:GP 1|0:0.00,0.99,0.01 ↑↑↑ hap1=ALT(T), hap2=REF(C) 1 768116 . C A . PASS . GT:GP 1|1:0.00,0.00,1.00 homozygous alt — same on both haplotypes 1 810000 . ACGT A . PASS . GT:GP 0|1:0.01,0.97,0.02 ← an indel: 3bp deletion on haplotype 2 ... ~6–10 million variants per donor across all chromosomes
Why phasing matters: Without phasing, we can't build two separate haplotype sequences — we'd know what variants exist but not which chromosome copy each belongs to. The HipSci BCFs are phased using SHAPEIT2, which gives us the |-separated genotypes that let us build hap1 and hap2 separately.

Build N-masked FASTA with bcftools consensus + bedtools maskfasta

We build a single personalized reference FASTA in two passes, then prepare a SNPsplit input file for later haplotype splitting.

Reference (GRCh37):  ...ACGTACGATACGTACGCTACGT...
                                ↑ pos 752566    ↑ pos 754192
Variant at 752566:  1|1  (homozygous alt = G on both copies)
Variant at 754192:  0|1  (heterozygous: hap1=REF=C, hap2=ALT=T)

Pass 1 — apply hom-alt SNPs (donor agrees on both copies):
After:   ...ACGTACGGTACGTACGCTACGT...   ← 752566 substituted to G

Pass 2 — N-mask het SNPs (donor's two copies disagree):
After:   ...ACGTACGGTACGTACGNTACGT...   ← 754192 replaced with N

The aligner now:  • at pos 752566: sees G — correctly matches reads with alt allele
                  • at pos 754192: sees N — accepts any base equally (no allele preference)

🔵 Homozygous-alt site

Both copies carry the alt allele. We substitute directly: the personalized reference has the donor's actual base here. Both alleles are the same so there's no ambiguity.

🟡 Heterozygous site (N-masked)

The two copies carry different alleles. We write N — the aligner treats N as equally compatible with any base, so neither allele is preferred. WASP catches any residual bias.

HPSI0114i-bezi_1.Nmasked.fa FASTA
>1 ← standard chromosome names — no duplication, GTF annotations apply directly NNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNNN TAACCCTAACCCTAACCCTAACCCTAACCCTAACCCTAACCCTAACCCT ... [reference sequence at most positions] ... [at pos 752566: G (hom-alt substituted)] ... [at pos 754192: N (het site masked)] ... ~1.5–2.5M N positions per donor genome >2 ← chromosome 2 ...

We also prepare a SNPsplit input file listing all het SNP positions with their phasing. This is used in Step 5c to assign reads to haplotype 1 or haplotype 2 after alignment.

HPSI0114i-bezi_1.snpsplit_input.txt TSV
CHROM POS REF ALT GT ────────────────────────────────────── 1 754192 C T 0|1 ← hap1=C(REF), hap2=T(ALT) 1 768116 T G 1|0 ← hap1=G(ALT), hap2=T(REF) ... ~1.5–2.5M het SNPs per donor
Why not build two full haplotype FASTAs? Short reads (~150 bp) that don't overlap any variant site would map equally well to both haplotypes, producing duplicate alignments throughout the genome (multi-mapping explosion). The N-masked single-reference approach avoids this — all reads align to one genome, and SNPsplit assigns them to a haplotype only when they actually overlap a het SNP (Step 5c).

Build the STAR index for this donor

STAR pre-processes the N-masked FASTA into a searchable index. Because the sequence differs per donor, each donor needs their own STAR index. At ~30 GB (same as a standard haploid index), we build it, align all cells for that donor, then delete it before moving to the next donor.

One standard GTF — no duplication needed. The N-masked FASTA uses the same chromosome names as GRCh37 (1, 2, ..., X), so the GENCODE v19 GTF applies directly without any modification. This is one of the key advantages of the N-masked approach over a diploid-chromosome reference.
4
Download the scRNA-seq CRAMs

ENA hosts 46,468 CRAM files for this dataset — one file per cell. We only download cells from the 104 open-access donors (~31 GB of the total ~38 GB).

A CRAM is a compressed alignment format. It stores reads as differences from a reference genome rather than storing the full sequence of each read. This makes files ~3× smaller than the equivalent FASTQ, but requires the reference genome to decode them.

Important: The CRAMs were aligned with BWA-MEM, a DNA aligner that doesn't understand RNA splicing. We can't use these alignments directly — we'll discard the alignment information and re-align with STAR. The CRAMs are essentially just a very compressed way to store the raw reads.
19776_1#1.cram (decoded to SAM view) BINARY
CRAM is binary — this is what it looks like when decoded with samtools view: ────────────────────────────────────────────────────────────────────────────────── ERR1209149.1 0 1 752560 60 75M * 0 0 ACGTACGTACGTACGTACGTACGTACGTACGTACGTACGTACGTACGT IIIII... ↑↑↑ CIGAR: 75M = 75 bp match (no splicing detected — BWA-MEM missed the exon junction) ERR1209149.2 0 1 802340 60 75M * 0 0 GCTAAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCT IIIII... This read spans an exon junction but BWA-MEM just soft-clips it. STAR will correctly map it as e.g. 35M1765N40M (35bp exon1 + 1765bp intron + 40bp exon2)
5
Align each cell and produce BigWigs

5a. CRAM → FASTQ samtools collate + samtools fastq

We strip the (incorrect BWA-MEM) alignment and recover just the raw read sequences. A FASTQ file has four lines per read: a name, the DNA sequence, a separator, and per-base quality scores.

19776_1#1_R1.fq.gz FASTQ
@ERR1209149.1 ← read name ACGTACGTACGTACGTACGTACGTACGTACGTACGTACGTACGTACGTACGTACGTACGTACGTACGTACGT ← 75 bp read sequence + IIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIII ← Phred quality (I = Q40, very high confidence) @ERR1209149.2 GCTAAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCTAGCT + HHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHHH ← H = Q39 ... thousands of reads per cell

5b. STAR alignment to N-masked reference + WASP bias correction

STAR aligns each read against the N-masked single reference. At het sites (N positions), any base is accepted equally. We run STAR with WASP mode enabled: for every read overlapping a het SNP, STAR internally flips the allele and re-aligns — if the read maps differently with the flipped allele, it was reference-biased and gets flagged for removal. This is the key step that removes alignment bias at het sites.

Read sequence:   ACGTACGTGTACGTACG CTACGTACGT...
                          ↑ het SNP at pos 754192 (REF=C, ALT=T → masked to N)

N-masked ref:    ACGTACGTNTACGTACG CTACGT...   ← accepts G equally
→ Read maps here; WASP flips allele to C, tries to re-align
→ Re-alignment lands at same position → read is unbiased → PASS (vW:i:1)

A biased read carrying REF allele C at a het site:
First alignment:   ACGTACGTCTACGT...  maps fine (N accepts C)
WASP flips to T:   ACGTACGTTTACGT...  maps to a different location
→ Result differs → read was reference-biased → FAIL (vW:i:2 or higher) → discarded

A junction-spanning read:  EXON1_SEQ ─── EXON2_SEQ
N-masked ref:              EXON1     intron     EXON2   ← STAR finds "35M1765N40M"

The result is a BAM file. Every read carries a vW tag indicating its WASP status. Only reads tagged vW:i:1 (unbiased) proceed to the next step.

NH:i:1 vW:i:1 ↑↑↑↑↑ WASP: unbiased → keep ERR1209149.2 0 1 802340 255 35M1765N40M * 0 0 GCTAAGCT... NH:i:1 vW:i:1 CIGAR: 35M = 35bp exon1, 1765N = intron skipped, 40M = exon2 ERR1209149.3 0 1 752560 255 75M * 0 0 ACGTACGTCTACGT... NH:i:1 vW:i:2 ↑↑↑↑↑ WASP: allele flip changed mapping → discard

5c. WASP filter + SNPsplit haplotype assignment

Two sequential steps: first discard biased reads using the WASP tag, then assign the remaining reads to haplotype 1 or haplotype 2 using the phasing information from the VCF.

WASP filter: Keep only reads tagged vW:i:1 (WASP pass) — these are reads where flipping the allele did not change the mapping, i.e., reads that would have been found regardless of which allele is in the reference.

SNPsplit: For each WASP-passing read that overlaps a het SNP, look up which allele the read carries and compare it to the phased VCF. If the read carries the 0|1 left allele → haplotype 1 BAM. If it carries the right allele → haplotype 2 BAM. Reads that don't overlap any het SNP (~60–80% of reads) go to an unassigned BAM.

wasp_filtered.genome1.bam (hap1)

Reads carrying the phased hap1 allele at a het site. These represent RNA transcribed from chromosome copy 1. Standard chromosome names (1, 2, ..., X) — no renaming needed.

wasp_filtered.genome2.bam (hap2)

Reads carrying the phased hap2 allele at a het site. RNA from chromosome copy 2. Same coordinate space as hap1 — both map to the N-masked hs37d5 reference.

60–80% unassigned reads are expected. Most Smart-seq2 reads (~75 bp) don't overlap any het SNP. These reads provide diploid-summed coverage but cannot be haplotype-assigned. The h1 and h2 BigWigs will be sparser than the diploid-summed track; this is a known limitation of short-read haplotype phasing.

5d. BAM → BigWig deeptools bamCoverage

The final step converts each BAM to a BigWig — a compact binary format that stores a single number (RNA coverage) for every base position in the genome.

We produce three BigWig files per cell: .h1.bw and .h2.bw from the SNPsplit haplotype BAMs, and .diploid.bw from all WASP-passing reads (including unassigned). Coverage is normalized to CPM (counts per million mapped reads) so cells sequenced to different depths are comparable. For most DNA LM training, the diploid BigWig is the primary signal; the haplotype BigWigs enable allele-specific analysis.

Each training example pairs a window of DNA sequence with the RNA coverage that sequence produced. The diploid BigWig is the primary training signal; the haplotype BigWigs (h1, h2) give additional signal for allele-specific effects when reads can be phase-assigned. What this teaches the model: Across 104 donors × ~36K cells, the model sees the same genomic region with slightly different sequences (each donor's N-masked personalized genome) and slightly different RNA output. It can learn that a particular variant (e.g., a SNP in a promoter) is associated with higher or lower expression — the signal that comes from genetic variation across individuals. When haplotype-resolved h1/h2 BigWigs are available, the model can additionally learn allele-specific effects within a single donor.
6
Processing order and disk management

Each donor's STAR index is ~30 GB. Building all 104 at once would require 3+ TB. Instead, process one donor at a time and delete the index when done.

Step Time Disk needed Delete after?
Extract per-donor VCF (Step 3b) ~5 min ~500 MB Yes (keep final .vcf.gz)
Build N-masked FASTA + SNPsplit input (Step 3c) ~10 min ~3.5 GB Optional (keep for re-runs)
Build STAR index (Step 3d) ~30 min ~30 GB Yes — delete after all cells aligned
Download CRAMs for donor (Step 4b) varies ~300 MB (avg ~350 cells) Yes (after FASTQ produced)
Align all cells for donor (Step 5) ~20 hrs on 4 cores ~100 MB/cell transient Keep BigWigs only
Final output per cell: three BigWig files — CELL_ID.h1.bw, CELL_ID.h2.bw, and CELL_ID.diploid.bw — each ~1–5 MB. Total for 104 donors × ~350 cells × 3 tracks ≈ ~109,000 BigWig files, ~200–600 GB total.
