From raw reads to base-pair coverage:
what data exists and how to use it

A plain-English guide to the data formats, tools, and pipeline choices for getting BigWig coverage tracks from Cuomo et al.'s Smart-seq2 scRNA-seq data.

⚠️
The Zenodo files are not what we need The Zenodo record (zenodo.org/record/3625024) has gene-count matrices: one number per gene per cell. All the spatial, splicing, and allele information is gone. For a DNA language model that needs base-pair coverage, we must go back to the raw reads.

Why the aligner matters: splicing

RNA is processed before it leaves the nucleus — "introns" are cut out, leaving only "exons." A read sequenced from mRNA can therefore span a junction between two exons that are far apart on the genome. Only an RNA-aware aligner can handle this.

File formats & tools explained

Every format and program you'll encounter when working with this data.

📄
FASTQ
.fastq.gz

The raw output from the sequencer. Each entry is one short DNA read (~150 bp) with a quality score per base. Think of it as a text file of millions of short DNA fragments, each paired with a confidence score.

This is the "purest" form of the data — no assumptions about where the reads come from.

Raw reads
🗜️
CRAM
.cram / .crai

A compressed alignment file. It stores the same information as a BAM file (where each read maps on the genome) but uses the reference genome to compress it, making it ~3× smaller.

The .crai is an index file — like a book index — allowing random access without reading the whole file.

Pre-aligned
📦
BAM
.bam / .bai

Like CRAM — a sorted, compressed alignment file storing where each read maps on the genome. Each row is a read with its genomic coordinates, mapping quality, and flags.

BAM is the standard input/output for most genomics tools.

Aligned reads
🚫
BWA-MEM
bwa mem

A DNA aligner — the standard tool for aligning whole-genome sequencing or exome data to a reference. It does not know that RNA has been spliced.

This is what Sanger used to produce the CRAM files. It was the lab's default pipeline regardless of library type — a mistake for RNA-seq data.

Wrong for RNA-seq
⭐
STAR
STAR

An RNA-aware splice junction aligner. STAR discovers exon-exon junctions in a first pass, then re-aligns all reads using this splice site map in a second pass ("2-pass mode"). It correctly places reads that span introns.

This is what the Cuomo paper itself used for analysis — but those BAM files were never deposited publicly.

Correct for RNA-seq
🐟
Salmon
salmon quant

A transcript quantifier. Salmon takes reads and a transcript reference and outputs one number per gene: an estimated read count. It is extremely fast and accurate for gene expression analysis.

Not what we want. Salmon discards the positional information entirely — it just counts, losing the base-pair coverage signal we need.

Loses position info
📊
BigWig / bedGraph
.bw / .bedGraph

A coverage track — one number per genomic base position representing how many reads overlap that position. This is the base-pair resolution signal we want for training the DNA language model.

bedGraph is the text version; BigWig is the compressed binary version. Both store the same information.

Base-pair resolution What we want
🔧
samtools fastq
samtools fastq

Extracts the original raw reads back out of a CRAM or BAM file, discarding the alignment information. The result is a FASTQ file — as if the alignment never happened.

This lets us re-use the CRAMs as a compact download format, then re-align with STAR instead of BWA-MEM.

CRAM → FASTQ
📈
bamCoverage
deeptools bamCoverage

Converts a BAM/CRAM file into a BigWig coverage track. Takes a sorted, indexed BAM and outputs one value per genomic bin (we want bin size = 1 bp for full resolution).

Also handles normalization (CPM, RPKM) so that cells with different sequencing depths are comparable.

BAM → BigWig
🛡️
WASP filtering
--waspOutputMode SAMtag

When a read overlaps a position where the donor differs from the reference genome, there is a risk that reads carrying the reference allele map better than reads carrying the donor's allele — "reference bias."

WASP fixes this by re-mapping each affected read with the alleles flipped. If the result changes, the read is flagged and discarded. Built into modern STAR.

Removes allele bias

Three ways to get coverage tracks

The options differ in bandwidth, correctness, and compute. The recommended path is Option A.

A
Re-align from CRAMs with STAR
Recommended

Download the CRAMs from ENA (~37.9 GB total), pipe each one through samtools fastq to recover the raw reads, then align with STAR in 2-pass mode. STAR can also emit bedGraph coverage directly with --outWigType bedGraph, skipping a separate bamCoverage step.

CRAMs are lossless containers of the original reads — using them saves ~3× bandwidth compared to downloading the separate FASTQ files. No information is lost by discarding the BWA alignment.

Pros
  • Correct splice-aware alignment
  • 3× smaller than downloading FASTQs
  • Can emit BigWig in one step
  • Embarrassingly parallelizable
Cons / caveats
  • ~10–30K CPU-hours total
  • Need hs37d5 reference to decode CRAMs
  • Need to re-align to GRCh38 yourself
# For each cell CRAM: samtools collate -uOn 128 cell.cram /tmp/tmp \ | samtools fastq -F 0xB00 -1 /dev/stdin -2 /dev/stdin - \ | STAR --runMode alignReads \ --genomeDir STAR_GRCh38 \ --outSAMtype BAM SortedByCoordinate \ --twopassMode Basic \ --outWigType bedGraph # base-pair coverage
B
Use the BWA-MEM CRAMs as-is
Quick but imperfect

Run deeptools bamCoverage directly on the existing CRAMs without re-aligning. Reads that span exon-exon junctions will be soft-clipped or missing, so the coverage at junctions is underestimated.

For a first exploratory pass, this may be acceptable — reads falling entirely within an exon will still map correctly. But it cannot be used for training a model that should learn about splicing.

Pros
  • Zero additional alignment compute
  • Good for quick exploratory checks
  • Exonic coverage is roughly correct
Cons / caveats
  • Splice junctions wrong/missing
  • Not suitable for final training data
  • Systematic bias against multi-exon reads
C
Download FASTQs from ENA, re-align
Also valid, more bandwidth

ENA also exposes the reads as FASTQ files (decoded from the CRAMs). You can download these directly and run STAR on them — same outcome as Option A, but you skip the CRAM decode step. However, these FASTQs are ~3× larger to download than the CRAMs.

Useful if you don't want to deal with CRAM decoding or if you want to archive the FASTQs locally.

Pros
  • Correct splice-aware alignment (same as A)
  • Simpler pipeline (no CRAM decode)
  • Familiar format for most tools
Cons / caveats
  • ~3× more download bandwidth
  • Larger local disk footprint
  • No benefit vs Option A otherwise
💡
Scale estimate for the full dataset ~36,000 cells × ~300K read pairs each = ~10–30K CPU-hours of STAR alignment. Each per-cell BigWig at 1 bp resolution will be ~5–50 MB. Total storage for all 36K cells: roughly 200 GB – 2 TB depending on compression and binning. The pipeline is embarrassingly parallel — each cell can be processed independently.
