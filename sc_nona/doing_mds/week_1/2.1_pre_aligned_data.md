# 2.1 — Pre-aligned data for Cuomo et al. 2020 (PRJEB14362)

## Summary

**Yes, pre-aligned data exists** — all 46,468 ENA runs in PRJEB14362 ship as **coordinate-sorted, indexed CRAM files** (~37.9 GB total) under open access at `ftp.sra.ebi.ac.uk`. **But the alignment is BWA-MEM against hs37d5 (NOT splice-aware)**, because Sanger's default NPG pipeline treats all Illumina output the same regardless of library type. The STAR 2-pass and Salmon outputs that the Cuomo paper actually used for analysis are **not deposited anywhere publicly** — only the gene/transcript count matrices (Zenodo) and the Sanger BWA CRAMs (ENA) are released. **No BigWig / bedGraph / coverage tracks exist for this dataset.** Fastest path to true splice-aware base-pair coverage = run STAR on the CRAMs (skip cram2fastq, use `samtools fastq | STAR`) and emit per-cell BigWigs with `bamCoverage` / `STAR --outWigType bedGraph`.

## Findings per source

### 1) ENA PRJEB14362 — CRAM files (BWA-MEM, hs37d5)

This is the only pre-aligned public artifact. The `submitted_format` column for every run is `CRAM;CRAI`. Example URLs:

```
ftp.sra.ebi.ac.uk/vol1/run/ERR146/ERR1462940/19776_1#5.cram      (13 MB)
ftp.sra.ebi.ac.uk/vol1/run/ERR146/ERR1462940/19776_1#5.cram.crai (1.2 KB)
```

Header (`samtools view -H`) reveals the Sanger NPG pipeline:

- `@HD SO:coordinate` — coordinate-sorted
- `@PG ID:bwa' VN:0.7.13` with command line: `bwa mem -t 16 -p -T 0 .../1000Genomes_hs37d5/all/bwa0_6/hs37d5.fa` — **BWA-MEM, not splice-aware**
- Reference: `hs37d5.fa` (GRCh37 primary + decoys + EBV, the 1000-Genomes phase-2 reference)
- `bamstreamingmarkduplicates` ran → duplicates marked
- Library strategy = `RNA-Seq`, source = `TRANSCRIPTOMIC SINGLE CELL`, platform = `Illumina HiSeq 2000/2500`

**Implications for BigWig coverage:**

- Reads that span exon-exon junctions will be soft-clipped or fail to map; BWA-MEM cannot place split reads across introns. So pile-up coverage from these CRAMs **will systematically under-count junction-spanning reads** — exactly the splicing signal you wanted base-pair coverage to capture.
- However, the read count per cell is small (~300 K reads, ~40 Mb bases for ERR1462940) so the fraction of junction-spanning reads is modest; intronic/exonic pile-ups from BWA-MEM coverage are still meaningful for transcribed regions.
- A cleaner approach: use these CRAMs only as a fast FASTQ source (`samtools fastq`), then re-align with STAR.

### 2) EGA EGAS00001002278 / EGAD00001005741

768 CRAM files, 37.9 GB total, controlled access via DAC EGAC00001000205. Almost certainly **the same underlying CRAMs** as ENA, but staged behind a DAC because some samples (the disease/rare-genotype donors) require managed access. ENA hosts the open subset. **Provides nothing extra** (no BAM, no BigWig — same CRAM format per the EGA file-type listing).

### 3) GitHub `single-cell-genetics/singlecell_endodiff_paper`

Public, last touched 2020. Three dirs: `scrnaseq_preprocessing/` (Snakefile, ~68 KB), `merging_and_qc/` (Jupyter), `plotting_notebooks/` (Jupyter).

Pipeline inferred from Snakefile:

```
Sanger NPG CRAM (BWA-MEM, hs37d5)
   |
   v   samtools collate | samtools fastq      <-- key step: ignore BWA alignments, extract reads
FASTQ per cell
   |
   v   STAR 2-pass (genome dir: STAR_GRCh37.75_ERCC), Picard rgadd/dedup, GATK SplitNCigar/BQSR
STAR BAM (.2pass.Aligned.sortedByCoord.split.realigned.bqsr.bam)   <-- NOT deposited anywhere
   |
   v   GATK ASEReadCounter (for allele-specific expression)
   +   Salmon quant (against GRCh37.75 + ERCC92 tx fasta)             <-- output that ends up on Zenodo
   +   featureCounts (uniq + nonuniq gene counts)
```

The intermediate STAR BAMs (`.2pass.Aligned.sortedByCoord.split.realigned.bqsr.bam`) and Salmon `quant.sf` files are **scratch/private** at EMBL-EBI (`/hps/nobackup/...`); they were not uploaded to ENA, EGA, Zenodo, or HipSci FTP.

### 4) HipSci portal + FTP

- The portal page (web archive) describes **bulk RNA-seq** distribution: "Raw sequencing reads — Distributed in the cram file format" and "Splice-aware STAR alignment — Distributed in the bam file format. **We distribute one bam file per cell line.**" One BAM *per line* is per donor (collapsed across all bulk RNA-seq runs of that iPSC line); this is not a per-cell file for Smart-seq2 scRNA.
- The FTP `track_hub/hipsci_hub/hg19/trackDb.txt` contains only exome-seq BAM tracks. No RNA-seq, no scRNA, no BigWig.
- `data/rnaseq/release/20170127/` contains `.gene.cov` text files (bulk RNA-seq, per cell line, gene-level coverage only — not base-pair). Filename pattern (`HPSI0114i-bezi_1.hs37d5.bwa.realigned.star.markdup.rnaseq.20150415.gene.cov`) confirms Sanger does run STAR for bulk RNA-seq, but again — bulk-only deliverable.

### 5) Single Cell Expression Atlas / ArrayExpress / GEO

Not hosted in SCEA (catalog API returns 0 matches for PRJEB14362 / Cuomo / endoderm-iPSC at the scale of 36 K cells). Companion ChIP-seq is in NCBI (PRJNA593217). No reprocessed/standardized version exists.

## Recommendation: fastest path to base-pair coverage

Pre-aligned splice-aware data **does not exist publicly** for this dataset. The fastest path is:

### Option A (recommended) — re-align CRAMs with STAR

The CRAMs are ~37.9 GB compressed total, much smaller than the FASTQ-equivalent on ENA. Skip the FASTQ round-trip Cuomo et al. used — pipe `samtools fastq` directly into STAR:

```bash
samtools collate -uOn 128 input.cram /tmp/prefix \
  | samtools fastq -F 0xB00 -1 - -2 - - \
  | STAR --runMode alignReads --readFilesIn /dev/stdin \
         --genomeDir STAR_GRCh38_or_GRCh37 \
         --outSAMtype BAM SortedByCoordinate \
         --twopassMode Basic \
         --outWigType bedGraph    # <-- gives you per-base coverage directly
```

Note that CRAM decode requires the **hs37d5 reference**; the header URLs point to the Sanger Lustre paths, but the file itself is available at:
`ftp://ftp.1000genomes.ebi.ac.uk/vol1/ftp/technical/reference/phase2_reference_assembly_sequence/hs37d5.fa.gz`

For modern work you'd probably want to bypass hs37d5 entirely and re-align to GRCh38 — that's only useful if you `samtools fastq` the CRAM (which loses the alignment but recovers the raw read pairs perfectly, since CRAM is a lossless container of the BAM).

Compute estimate: STAR 2-pass on Smart-seq2 ~300 K reads/cell × 36 K cells is embarrassingly parallel and totals ~10–30 K CPU-hours on modern hardware — feasible on a cluster.

### Option B — accept BWA-MEM coverage from the CRAMs as-is

For a first pass you can pile-up the existing CRAMs with `samtools depth` / `bamCoverage`, accepting the splice-junction soft-clipping. This is wrong for splicing analysis but probably fine if you only want exonic per-base pile-ups (most reads will still place against the genome since BWA-MEM does soft-clip rather than reject). Time cost: zero alignment, just `samtools depth -a` or `deeptools bamCoverage --binSize 1` over each CRAM.

### Option C — use the FASTQs from ENA, re-align ourselves

ENA also exposes `fastq_ftp` (decoded back from the CRAMs). Same compute story as Option A, but loses the small bandwidth win of CRAM and you don't need the hs37d5 ref. Likely the most reproducible path if disk and bandwidth aren't tight.

### What I would do

Option A. The CRAMs save ~3x bandwidth and disk vs FASTQ, and `samtools fastq` from a CRAM is lossless. You get correct splice-aware coverage in one pass, and STAR's `--outWigType bedGraph` produces per-base tracks directly — no separate `bamCoverage` step. After STAR, convert to BigWig with `bedGraphToBigWig` for compact storage. Per-cell BigWigs at ~5–50 MB/cell × 36 K cells ≈ 200 GB – 2 TB total depending on bin size and compression.
