Source: Nair et al., "Nona: A unifying multimodal masking framework for functional genomics" (bioRxiv 2025.11.06.687036, posted Nov 8 2025), `/Users/gloria/dev/research/arc/sc_nona/papers/nona.pdf`. Page numbers below refer to PDF pages in that file.

## TL;DR up front

Nona does **NOT** ingest single-cell data in the scRNA / scATAC sense the user is asking about. It uses **bulk functional-genomics BigWig tracks** (DNase-seq, CAGE-seq, and one bulk ATAC-seq dataset) downloaded as already-processed signal tracks. The only "single-cell" connection is that ATAC-seq fragment files (a single-cell-ish format) are used in the genotyping experiment, but even there, Nona consumes bulk-aggregated base-resolution BigWigs, not per-cell data. There is no pseudobulking, no cell-type aggregation step, no per-cell tracks, no scRNA, no Hi-C, no ChIP-seq actually used (despite the cartoon in Fig. 1a showing ChIP). The paper does not describe a read-alignment pipeline at all in most cases: tracks are downloaded pre-processed from ENCODE/FANTOM. The one alignment step they touch is BAM-to-BigWig conversion for ATAC-seq using ChromBPNet's `reads_to_bigwig.py`.

## 1. Which functional/single-cell modalities does Nona actually incorporate?

Three concrete models are trained, each with a narrow modality slice (no scRNA, no scATAC, no Hi-C, no ChIP, no CAGE-on-single-cells):

- **Context-aware model (Fig. 2, Methods p. 21):** 7 K562 DNase-seq BigWigs + 2 K562 CAGE-seq BigWigs (human) + 4 mouse DNase-seq + 3 mouse CAGE-seq. All **bulk**. Tracks were chosen to overlap with Enformer's track set (p. 21, "Data" subsection).
- **Functional language model (Fig. 3, Methods p. 23):** a single **bulk GM12878 DNase-seq** track (ENCFF093VXI), plus a sequence-only baseline. No tracks beyond that.
- **Functional genotyping model (Fig. 4, Methods p. 26):** a single **bulk GM12878 ATAC-seq** track derived from ENCSR637XSC BAMs. AFGR ATAC-seq (DeGorter et al. ref 33) is used only for held-out evaluation, not training.

Fig. 1a shows DNase, ChIP, …, CAGE as illustrative track types; but per Methods, only DNase, CAGE, and (bulk) ATAC are actually used. ChIP-seq is not actually trained on anywhere in the paper.

**No single-cell modalities are incorporated.** The Discussion (p. 11) frames the ATAC-seq work as relevant to "single-cell applications" only because fragment files are common in scATAC pipelines — but the model itself trains on a single aggregated bulk BigWig.

## 2. Exact data preprocessing/alignment pipeline

### Reference genome
- **hg38** for human throughout (p. 22, TRIP-seq section: "extracted the seven promoter sequences from hg38"; p. 26, ATAC genotyping section: "the model was trained on the hg38 reference genome rather than GM12878's genome"; RepeatMasker file is `hg38.fa.out.gz`, p. 25).
- Mouse genome assembly is not explicitly stated — only that Enformer's mouse splits and tracks are reused (p. 21, "We used the same train, validation and test splits as Enformer for both human and mouse"). Enformer uses mm10, but Nona does not say this directly. **Gap.**
- No T2T.

### Read alignment tool
- **Not described for DNase/CAGE.** They download already-aligned, read-depth-normalized BigWigs from ENCODE (DNase) and unstranded summed BigWigs from FANTOM (CAGE) (p. 21). No mention of BWA/STAR/bowtie/chromap/cellranger.
- **For ATAC-seq:** they download BAM files from ENCODE (ENCFF440GRZ, ENCFF962FMH, ENCFF981FXV) — already aligned by ENCODE upstream — merge them, and convert with `reads_to_bigwig.py` from ChromBPNet, run inside Docker image `kundajelab/chrombpnet:latest` with the listed sha256 digest (p. 26, "Tracks and training regions"). So the explicit alignment tool used by Nona authors is **none** — they consume aligned BAMs from ENCODE.



### Reads → per-base signal
- **DNase-seq:** "read-depth normalized signal track BigWigs" downloaded directly from the ENCODE portal; replicates merged (p. 21).
- **CAGE-seq:** forward and reverse strand BigWigs from FANTOM, merged into unstranded tracks; replicates not considered (p. 21).
- **ATAC-seq (training):** BAM → base-resolution BigWig via `reads_to_bigwig.py` (ChromBPNet). "The created BigWig files contain base-resolution signals where the value at a specific base corresponds to the number of reads in the BAM file that either start or end at that position, after accounting for Tn5 shift" (p. 26, AFGR section). So these are **Tn5 insertion-count tracks**, base-resolution, per-position read counts.

### Bin size / resolution
- Tracks are **base-resolution as input** (one value per bp) throughout (e.g. p. 17 Methods "Input and output": "Nona operates on masked DNA sequence and base-resolution tracks", L_in × T matrix).
- The functional language model and the genotyping model both operate at base resolution with `n_reductions=0` (no down/up-sampling) (p. 18, p. 23, p. 27).
- The context-aware model has `n_reductions=7` (U-Net style); input 196,608 bp, output cropped to 114,688 bp or 4096 bp (p. 21). Downstream metrics are aggregated to **128 bp resolution** to match Enformer's output binning (p. 6, p. 22).

### Normalization
- Tracks are downloaded already "read-depth normalized" from ENCODE (p. 21). No further per-sample normalization described for DNase/CAGE.
- The data loader applies **`log1p`** to track values before feeding them in (Methods, p. 19: "Before model input, track values are transformed using log1p").
- Predicted/target tracks are clipped to a fixed range before loss (typically between −10 and ln(5000), per Methods p. 18 Architecture and p. 21 — clipping bound 8.517 ≈ ln(5000)).
- Loss is **Poisson NLL** on log-scale predictions for tracks (Methods p. 19 "Loss"; code snippet uses `poisson_nll_loss(..., log_input=True)`).
- For ATAC genotyping, AFGR BAMs are subsampled (samtools view -s) to 1M/2M/.../10M reads to study depth robustness (p. 26). No CPM/RPKM/quantile/arcsinh.

### How tracks are aligned to the input DNA the model sees
- Tracks are **parallel tracks stacked alongside the DNA**, both indexed by the same genomic coordinate (Fig. 1a,b; Methods p. 17 "Input and output").
- Input is an `L_in × 4` one-hot DNA matrix and an `L_in × T` track matrix, plus two binary masks `L_in × 4` and `L_in × T`. Total channel count is `2 × (4 + T)`. Sequence and tracks are concatenated with their masks and passed through **separate convolutional encoders**; encoder outputs are **summed before the transformer** (Methods p. 18).
- This is **not MSA-style** (no second axis of homologous samples). It's the Borzoi/Enformer-style track-as-channels layout, with masking added.

## 3. How is the aligned signal *used* in the architecture?

- **Hybrid convolution-transformer U-Net** inspired by Borzoi (Methods p. 18). Separate conv encoders for sequence and tracks → encoder outputs summed → transformer blocks (x-transformers, flash attention, rotary positional embeddings) → upsampling decoder with skip connections.
- For the **language-model / genotyping models**, `n_reductions=0`: no U-Net; pure conv+transformer at base resolution (p. 18, p. 23, p. 27).
- Tracks function as **both inputs and outputs**. The model has a head producing `L_out × T` track logits (log-scale, Poisson loss) and a head producing `L_out × 4` base logits (cross-entropy). A masking configuration (Fig. 1c) determines which sub-task is being run: sequence-to-function (mask all tracks), context-aware (mask central 4096 bp of tracks), track-to-track, functional language modeling (sequence partially masked, tracks unmasked), etc.
- This is closer to **Enformer's multi-track output head + masked-modality input conditioning** than MSA cross-attention. There is no explicit cross-attention between sequence and tracks — they are fused by summation after the conv encoders.
- Output values are clipped to (−10, 8.517) for stability (Methods p. 21).

## 4. Datasets, cell types, and number of cells

There are no "cells" in the single-cell sense — only bulk samples and one cohort of bulk ATAC-seq per individual. Specific accessions (p. 21, 23, 26):

- **Human DNase-seq (context-aware), K562:** ENCFF899YDP, ENCFF515UNC, ENCFF708UIS, ENCFF413AHU, ENCFF868NHV, ENCFF565YDB, ENCFF971AH0 (7 tracks).
- **Human CAGE-seq, K562:** CNhs11250, CNhs12336 (2 tracks, FANTOM).
- **Mouse DNase-seq:** ENCFF990AT0, ENCFF223QRV, ENCFF155MQS, ENCFF014NWA (4 tracks; 3 MEL, 1 leukemia stem cells).
- **Mouse CAGE-seq:** CNhs12202, CNhs12198, CNhs12551 (3 tracks; hematopoietic/myeloid progenitor analogs of K562).
- **Functional language model:** GM12878 DNase-seq BigWig ENCFF093VXI; peaks ENCFF5880CA; GC-matched non-peak regions sampled via `get_gc_matched_intervals` in gReLU (p. 23). Splits: chr1 test, chr8+chr10 validation, rest training.
- **Functional genotyping training:** GM12878 ATAC-seq, ENCSR637XSC, BAMs ENCFF440GRZ + ENCFF962FMH + ENCFF981FXV merged (p. 26).
- **Genotyping evaluation:** AFGR ATAC-seq for 100 individuals (DeGorter et al., ref 33), downloaded from ENCODE Jan 2025 (p. 26). 1000 Genomes high-coverage VCF for 3202 individuals as candidate genotypes; 4756 common SNPs (UCSC `snp151Common`) within the top-1000 GM18498 LCL ATAC-seq peaks (ENCFF717TCQ) (p. 26).
- **TRIP-seq evaluation (downstream):** Leemans et al. 2019 (7 promoters) and Hong et al. 2022 datasets (p. 22).

No GTEx, Tabula Sapiens, HCA, 10x Multiome, or HCL. No cell counts because no single-cell data are used.

## 5. Single-cell vs bulk; sparsity / pseudobulking / per-cell tracks

Explicit: **all training tracks are bulk.** There is no sparsity handling, no pseudobulking step, no cell-type aggregation, and no per-cell tracks anywhere in Nona's pipeline.

The closest single-cell touchpoint is in the framing of the genotyping experiment (p. 8, "Functional genotyping"): the authors note that ATAC-seq fragment files "have become a standard format for sharing data, particularly in single-cell applications" and that those fragment files can be "readily converted into base-resolution BigWig tracks". But the actual training data are bulk GM12878 ATAC-seq, and the AFGR evaluation samples are bulk per-individual ATAC-seq, not scATAC. So the privacy-risk claim is demonstrated only on bulk data, even though the authors argue it generalizes to scATAC repositories (Discussion, p. 11).

The Discussion also explicitly suggests that "larger Nona models jointly trained across multiple masking configurations could learn richer shared representations" (p. 12) — but no such larger or multi-cell-type model is built here.

## What's missing / gaps for someone wanting to adapt this to scRNA

- **No mouse genome assembly named.** Likely mm10 because Enformer splits are reused, but not stated.
- **No description of how BAMs were originally aligned** (ENCODE's standard pipeline is implicit — typically bowtie2 for ATAC and STAR/TopHat for CAGE — but Nona authors do not run alignment themselves).
- **No discussion of strandedness for RNA-like tracks.** CAGE is explicitly converted to unstranded by summing strands (p. 21). No strategy is given for stranded RNA-seq.
- **No splicing / junction handling.** All tracks are coverage-style BigWigs; no junction tracks, no transcript-level structure.
- **No per-cell or per-individual track stacking** (i.e., no MSA-style vertical axis). Tracks `T` is a fixed small number of bulk channels.
- **No normalization beyond ENCODE's depth normalization + log1p.** No CPM, RPKM, arcsinh, quantile.
- **No description of how variation across individuals is handled at the input level.** AFGR samples are evaluated by reusing the GM12878-trained model with no per-sample fine-tuning or read-depth normalization (p. 28); the model "relies primarily on base-resolution profile shape rather than read depth."
- **No promoter–enhancer pairing supervision.** Context-aware mode provides flanking-track context but no explicit pairing or contact data.

So Nona is, in practice, an Enformer/Borzoi-style sequence + bulk-BigWig model with a unified masking interface — it does not give you a worked recipe for ingesting scRNA, and any sc-extension would have to be designed from scratch (pseudobulk-by-cell-type → BigWig per channel is the most natural drop-in suggested by the architecture).
